## 1. 系统总体架构

### 1.1 架构概览

CogniLink 采用前后端分离的三层架构，基于 PostgreSQL + pgvector 作为统一存储层，实现向量检索与关系数据的一体化管理。

```
┌─────────────────────────────────────────────────────┐
│              前端 (Next.js 16 + TypeScript)           │
│  App Router · Tailwind CSS · Zustand · ECharts       │
│  认证中间件 · SSE 流式渲染 · 8-bit 像素 UI           │
├─────────────────────────────────────────────────────┤
│              后端 (FastAPI + SQLAlchemy Async)        │
│  API 路由层 → Service 业务层 → Model 数据层          │
│  LiteLLM 统一调用 · LangGraph 多 Agent · pgvector    │
├─────────────────────────────────────────────────────┤
│              数据库 (PostgreSQL 16 + pgvector)        │
│  13+ 张核心表 · 向量列 · 全文检索 · 级联删除         │
└─────────────────────────────────────────────────────┘
```

### 1.2 技术栈约束

| 层次 | 技术 | 版本/约束 |
|---|---|---|
| 前端框架 | Next.js (App Router) | ^16.x |
| 前端语言 | TypeScript (strict mode) | 禁用 `any` |
| UI 组件 | Radix UI + Tailwind CSS + shadcn/ui | Tailwind v4 |
| 状态管理 | React hooks + Zustand | 优先 hooks |
| 图表可视化 | ECharts | ^6.x |
| 后端框架 | FastAPI | 全异步 async/await |
| ORM | SQLAlchemy (Async) | 异步 session，分层架构 |
| 数据库 | PostgreSQL + pgvector | docker-compose 启动 |
| 认证 | PyJWT + bcrypt | JWT + cookie 双写 |
| AI | OpenAI / DeepSeek / GLM / LiteLLM | 统一调用层 |
| 本地嵌入 | FlagEmbedding (BGE-M3) | 无需 API key |
| 多 Agent | LangGraph + langgraph-checkpoint-postgres | 状态持久化 |

### 1.3 后端分层架构

后端严格遵循四层分离原则：

- **`api/`** — 路由接口层：接收 HTTP 请求，Pydantic 校验参数，调用 Service 层处理业务，抛出 HTTPException
- **`services/`** — 业务逻辑层：模块级单例模式（`*_service = ServiceClass()`），封装所有业务逻辑与算法
- **`models/`** — 数据模型层：SQLAlchemy ORM 模型（`database.py`）+ Pydantic 请求/响应模型（`schemas.py`）
- **`core/`** — 基础设施层：配置管理（`config.py`）、异步数据库引擎（`database.py`）、JWT 认证与权限依赖（`security.py` + `dependencies.py`）

|---|---|---|

## 2. 认证与权限系统

### 2.1 设计方案

采用 JWT（JSON Web Token） + bcrypt 密码哈希的认证方案，前端通过 localStorage + cookie 双写存储 Token。

### 2.2 核心组件

- **`core/security.py`** — 密码哈希与 Token 签发
  - `get_password_hash()`: bcrypt 加盐哈希
  - `verify_password()`: bcrypt 验证
  - `create_access_token()`: PyJWT 签发 HS256 Token，有效期 7 天

- **`core/dependencies.py`** — FastAPI 依赖注入链
  - `get_current_user`: 解析 Bearer Token → 查询 User 表 → 返回 User 对象
  - `get_admin_user`: 在 `get_current_user` 基础上校验 `role == "admin"`
  - `get_teacher_or_admin_user`: 校验 `role in ("admin", "teacher")`

### 2.3 认证流程

```
用户登录 → POST /api/auth/login
  → bcrypt 验证密码
  → 签发 JWT (sub=username, role=role, exp=7d)
  → 返回 {access_token, token_type, user}

前端接收 Token
  → localStorage.setItem("cognilink_token", token)
  → document.cookie = "cognilink_token=..." (供 middleware SSR 拦截)
  → 后续所有请求通过 getAuthHeaders() 携带 Authorization: Bearer <token>
```

### 2.4 前端认证中间件

`frontend/middleware.ts` 在 Next.js 服务端拦截所有请求：
- `/login` 路径免认证直接放行
- 其他路径检查 `cognilink_token` cookie，无 Token 则重定向到 `/login`
- 静态资源（`_next/static`、图片等）被排除

### 2.5 角色体系

| 角色 | 权限 |
|---|---|
| `student` | 学习、练习、对话、查看个人画像 |
| `teacher` | student 权限 + 查看学员列表与画像 |
| `admin` | 全部权限 + 管理后台（用户/文档/题库/Agent 管理） |

首个注册用户自动成为 `admin`（通过查询 User 表是否为空判定）。

### 2.6 API 接口

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/auth/register` | 注册（首个用户自动 admin） |
| POST | `/api/auth/login` | 登录 |
| GET | `/api/auth/me` | 获取当前用户信息 |

## 3. AI 导师对话系统

### 3.1 设计方案

对话系统支持三种模式：基础对话、RAG 增强对话、多 Agent 协同对话。所有对话通过 SSE（Server-Sent Events）流式输出，实现逐字渲染效果。

### 3.2 三种对话端点

| 端点 | 模式 | 说明 |
|---|---|---|
| `POST /api/chat` | 基础对话 | SSE 流式，Agent 路由，对话持久化 |
| `POST /api/chat/rag` | RAG 增强 | SSE 流式，四大特性开关（RAG/记忆/工具/本地嵌入） |
| `POST /api/chat/graph` | 多 Agent | SSE 事件流，LangGraph 工作流编排 |

### 3.3 四大特性开关

`RAGChatRequest` 定义了四个布尔开关，由前端设置页控制：

| 开关 | 字段 | 功能 |
|---|---|---|
| 知识库检索 | `use_rag` | 对话前检索文档向量，注入上下文 |
| 长期记忆 | `use_memory` | 对话前检索相关记忆，对话后自动提取记忆 |
| Agent 工具 | `use_tools` | 启用 OpenAI function calling（计算器/搜索/日期） |
| 本地嵌入 | `use_local_embedding` | 使用 Ollama BGE-M3 而非 API 嵌入 |

### 3.4 LLM 统一调用层（`llm_service.py`）

`LLMService.stream_chat()` 是核心入口：

1. **构建 system prompt**：按优先级注入 Agent 人格 prompt → RAG 检索上下文 → 记忆上下文
2. **构建 messages 列表**：system + 对话历史 + 当前用户消息
3. **API Key 回退**：前端未配置时回退到后端 `.env` 中的 DeepSeek 系统级密钥
4. **LiteLLM 调用**：使用 `acompletion` 异步流式调用，当有自定义 `base_url` 时添加 `openai/` 前缀路由
5. **Tool Calling 循环**：当启用 `use_tools` 时，累积流式分片中的 `tool_calls`，执行工具后将结果追加到 messages，最多 5 轮循环

### 3.5 Agent 自动路由（`agent_service.py`）

- **手动指定**：用户在前端选择 `humor`/`academic`/`coach` 或 UUID
- **自动路由**：LLM 分类用户消息的学习场景，路由到对应风格导师
- **认知状态注入**：查询用户知识状态（薄弱点、已掌握），注入到 system_prompt 实现个性化教学

三种导师风格：

| 角色类型 | 像素名 | 教学风格 |
|---|---|---|
| `humor_mentor` | 小柴 | 幽默风，用热梗和类比讲解概念 |
| `academic_mentor` | 小鹰 | 学术风，深度原理体系构建 |
| `coach_mentor` | 小铁 | 实战风，代码实操面试准备 |

### 3.6 流式响应处理

后端通过 `StreamingResponse` 返回 `text/event-stream`，设置禁用缓冲的响应头：
```python
STREAM_HEADERS = {
    "Cache-Control": "no-cache",
    "X-Accel-Buffering": "no",
    "Connection": "keep-alive",
}
```

前端 `useChat` hook 使用 `ReadableStream` + `TextDecoder` 逐 chunk 读取，30ms 节流更新 UI（约 33fps），`stream: true` 保留跨 chunk 多字节字符状态避免中文乱码。

## 4. 蜂巢技能树与自适应学习

### 4.1 设计方案

蜂巢技能树是 Dashboard 的核心组件，采用自研蜂巢布局算法，将 6 大领域的知识节点按错落网格排列，通过 SVG 连线展示依赖关系。

### 4.2 自研蜂巢布局算法（`skill-tree.tsx`）

布局核心逻辑：
1. **按分类分组**：6 大领域（programming/dsa/organization/os/network/database）各自一行
2. **行内排序**：每组内按 `pagerank_weight` 降序排列，保证重要性高的节点靠前
3. **蜂巢错落**：奇数行水平偏移半列宽度，形成蜂巢错落效果
4. **坐标计算**：行间距 130px，列间距 140px，每组首节点 x 坐标由前一组末尾节点决定

### 4.3 SVG 连线机制

| 关系类型 | 连线样式 | 说明 |
|---|---|---|
| `requires` | 实线箭头 | 前置依赖，必须先掌握源节点 |
| `extends` | 虚线 | 拓展关系，非强制前置 |
| 已点亮通路 | 分类色 | 用户已点亮的依赖路径 |
| 锁定通路 | 灰色 | 前置未满足的锁定路径 |

### 4.4 前置依赖解锁

节点解锁逻辑由后端 `knowledge_service.recommend_learning_path()` 实现：
- 仅推荐前置已全部点亮的未点亮节点
- 薄弱点加权：有 proficiency 但未点亮的节点优先推荐巩固
- 兜底策略：新用户无状态时退化为按 PageRank 重要性推荐

### 4.5 分类配色

六大领域采用高对比度糖果色系：
- 柠檬绿 — 编程开发基础
- 玫瑰红 — 数据结构与高级特性
- 蔚蓝 — 面向对象与系统架构
- 活力橙 — 并发编程与操作系统
- 浆果紫 — 网络编程与联机服务
- 薄荷绿 — 数据工程与持久化

## 5. 知识图谱（PageRank + 学习路径）

### 5.1 设计方案

知识图谱通过 ECharts force 力导向布局可视化，节点大小按 PageRank 权重映射（16-40px），支持点亮机制和自适应路径推荐。

### 5.2 图数据组装（`knowledge_service.get_graph_data`）

后端返回三部分数据：
1. **nodes** — 学习路线种子节点（`source='learning_path'`），含点亮状态、熟练度、学习时长
2. **relations** — 知识关系（仅保留两端都在学习路线子图中的边）
3. **stats** — 总节点数、已点亮数、按分类的点亮统计

支持按 `knowledge_base_id` 过滤特定知识库的专属图谱。

### 5.3 PageRank 算法（`knowledge_service.compute_pagerank`）

标准 PageRank 迭代算法实现：

```
PR(n) = (1-d)/N + d * (Σ PR(m)/outDeg(m) + dangling_sum/N)

参数:
  d = 0.85 (阻尼系数)
  max_iter = 100 (最大迭代次数)
  tol = 1e-6 (收敛阈值)
  N = 节点总数
```

算法特点：
- **纯结构计算**：不融合历史权重，避免重复计算导致分数衰减
- **悬挂节点处理**：无出度的节点 PR 均分给全图
- **纯算法函数**：`_pagerank_iterate` 不涉及 DB，便于单元测试
- **持久化**：计算后写回 `KnowledgeNode.pagerank_weight`

### 5.4 自适应路径推荐（`recommend_learning_path`）

推荐策略：
1. 加载学习路线节点、关系、用户状态
2. 优先用已持久化的 `pagerank_weight`，全 0 则现场计算
3. 筛选可推荐节点（未点亮 + 前置 `requires` 全满足）
4. 薄弱点加权：`score = pagerank * (1 + proficiency)`
5. 兜底：候选为空时退化为按重要性推荐
6. 按 score 降序取 Top-N

### 5.5 点亮机制（`apply_evaluation_result`）

评测后联动更新用户知识状态：
- **通过**（status=passed 或 score>=60）：点亮节点（`is_lighted=1`），永久标记
- **未通过但已点亮**：不撤销成就，标记 `already_lighted`
- **未通过且未点亮**：记录 proficiency 作为薄弱点，供路径推荐
- **熟练度取最高**：`proficiency = max(history, score/100)`，不因一次低分回退

### 5.6 双来源节点

| 来源 | 字段值 | 说明 |
|---|---|---|
| 学习路线种子 | `learning_path` | 蜂巢图/图谱展示的核心节点 |
| 文档自动提取 | `extraction` | AI 从上传文档中提取的概念节点 |

蜂巢图和图谱仅展示 `learning_path` 节点，`extraction` 节点服务于 RAG 问答。

### 5.7 API 接口

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/knowledge/graph` | 获取图数据（nodes + relations + 用户状态） |
| GET | `/api/knowledge/nodes/{node_id}/labs` | 获取节点关联的实验列表 |
| POST | `/api/knowledge/pagerank` | 重算全图 PageRank 权重 |
| GET | `/api/knowledge/recommend` | 获取个性化学习路径推荐 |

## 6. 在线练习系统（5 种题型）

### 6.1 设计方案

练习系统支持 5 种题型，3 种出题模式，评测结果联动知识图谱点亮机制。

### 6.2 五种题型

| 题型 | 代号 | 评测方式 | 前端渲染 |
|---|---|---|---|
| 代码题 | `code` | LLM 评测 | VS Code 风格编辑器 |
| 选择题 | `quiz` | 程序判分 | 单选即时反馈 |
| 匹配题 | `match` | 程序判分 | 左右连线匹配 |
| 排序题 | `arrange` | 程序判分 | 拖拽排序代码行 |
| 填空题 | `fill` | 程序判分 | 代码填空 |

### 6.3 三种出题模式

| 模式 | 说明 | 数据来源 |
|---|---|---|
| `system` | 内置题库 | Lab 表（管理员创建/导入） |
| `dynamic` | AI 针对薄弱节点出题 | `POST /api/labs/generate` 实时生成 |
| `collection` | 收藏夹 | UserCollectionExercise 表 |

### 6.4 AI 动态出题（`evaluation_service.generate_targeted_exercise`）

出题流程：
1. **节点选择**：未传 `node_id` 时自动取推荐薄弱节点或指定科目下最薄弱节点
2. **画像聚合**：构建用户学习画像上下文（学习阶段/科目状态/全局薄弱领域）
3. **RAG 检索**：获取该节点关联的文档分块作为知识参考
4. **LLM 生成**：调用 LiteLLM，注入节点信息 + 画像 + 知识上下文，按题型生成结构化 JSON
5. **虚拟 Lab**：生成的练习用 `dynamic-` 前缀 ID 标记，不持久化，前端据此走即时评测

### 6.5 评测引擎（`evaluation_service.py`）

#### 代码评测（`evaluate_code_submission`）
- 构建 Prompt（题目信息 + 测试用例 + 用户代码）
- 调用 `_call_llm_json` 获取 JSON 格式评测结果
- 返回：`{status, score, feedback, issues, suggestions}`

#### 选择题判分（`evaluate_quiz_submission`）
- 程序化比对 `test_cases` 中的正确答案与用户答案
- 计算得分百分比，生成反馈

### 6.6 评测联动知识图谱

评测后调用 `knowledge_service.apply_evaluation_result`：
- 通过 → 点亮知识节点 + 提升熟练度
- 未通过 → 记录薄弱点 → 下次 AI 出题聚焦薄弱节点

### 6.7 API 接口

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/labs` | 获取 Lab 列表（按类型/节点/难度筛选） |
| GET | `/api/labs/{lab_id}` | 获取 Lab 详情 |
| POST | `/api/labs/{lab_id}/submit` | 提交答案并评测（创建 submission 记录） |
| POST | `/api/labs/generate` | AI 动态生成针对性练习 |
| POST | `/api/labs/evaluate-dynamic` | 动态练习即时评测（不创建记录） |
| GET | `/api/labs/{lab_id}/submissions` | 获取提交历史 |

## 7. RAG 检索增强

### 7.1 设计方案

RAG 系统采用混合检索策略：BM25 稀疏检索 + 向量稠密检索 + RRF（Reciprocal Rank Fusion）融合，提升检索质量。

### 7.2 混合检索（`rag_service.py`）

#### 向量稠密检索（`search_similar`）
1. 生成查询向量（`embedding_service.get_single_embedding`）
2. pgvector `cosine_distance` 排序检索
3. 过滤：仅返回用户私有文档 + 共享文档，排除禁用文档

#### BM25 稀疏检索（`_bm25_search`）
1. PostgreSQL 全文检索：`to_tsvector` + `plainto_tsquery` + `ts_rank`
2. 关键词匹配打分
3. 异常时回退到空列表，不影响向量检索

#### RRF 融合（`hybrid_search`）
```
RRF_K = 60  (标准 RRF 常数)
retrieval_k = max(limit * 3, 15)  (过量检索以获得更好融合效果)

score(chunk) = Σ 1/(RRF_K + rank_i)  对每个检索器
```

### 7.3 上下文构建（`get_context_for_query`）

检索结果按 RRF 分数排序，每块附加来源标注：
```
[来源: 文档名, 第X页]
<chunk_content>
```

### 7.4 文档访问控制

RAG 检索内置权限过滤：
- 用户只能检索自己上传的私有文档 + 系统共享文档
- 管理员禁用的文档（`is_active=0`）不参与检索
- 知识库隔离：支持按 `knowledge_base_id` 过滤

### 7.5 嵌入服务（`embedding_service.py`）

双嵌入模式：

| 模式 | 提供商 | 实现 |
|---|---|---|
| API 嵌入 | OpenAI/阿里/智谱/月之暗面 | `AsyncOpenAI` 直连（绕过 LiteLLM） |
| 本地嵌入 | Ollama BGE-M3 | `/api/embeddings` 端点，信号量并发限 10 |

智谱特殊处理：真 API 用原生 `zhipuai` SDK（JWT 认证），代理用 `AsyncOpenAI`。

## 8. 文档处理流水线

### 8.1 设计方案

文档处理采用异步后台流水线，支持多格式解析、语义分块、向量化存储、AI 知识提取全流程。

### 8.2 处理流程

```
上传文件 (POST /api/documents/upload)
  → 保存文件到 uploads/
  → 创建 Document 记录 (status="processing")
  → asyncio.create_task(_process_document_async)

_process_document_async:
  1. document_processor.create_job() — 创建进度追踪任务
  2. document_service.parse_document_structured() — unstructured 解析
  3. document_service.chunk_text_structured() — 语义分块
  4. embedding_service.get_embeddings() — 批量向量嵌入
  5. 插入 DocumentChunk 记录 (含 embedding + element_type + page_number)
  6. doc.status = "completed"
  7. knowledge_extraction_service.extract_nodes_from_document() — AI 知识提取
  8. knowledge_extraction_service.extract_book_nodes() — 提炼学习主线节点
```

### 8.3 PDF 解析回退链

6 级回退链，首个产出 >= 60 个有效字符的提取器终止链：

| 优先级 | 提取器 | 特点 |
|---|---|---|
| 1 | `pymupdf4llm` | 最佳，Markdown 输出保留结构 |
| 2 | `pypdf` | 可靠的文本提取 |
| 3 | `pymupdf` (fitz) | 通用 PDF 解析 |
| 4 | `pdfplumber` | 擅长表格提取 |
| 5 | `ocrmac` | macOS 原生 OCR（中英文） |
| 6 | `RapidOCR` | 跨平台 ONNX OCR |

### 8.4 语义分块（`chunk_text_structured`）

- **优先**：`unstructured.chunk_by_title` 按标题语义分块（需完整 ElementMetadata）
- **回退**：固定大小分块（1500 字符），合并相邻元素，保留 element_type 和 page_number
- **伪元素**：传统解析器输出转换为 `_PseudoElement`，通过标题模式识别分类

### 8.5 进度追踪（`document_processor.py`）

内存级后台任务追踪器：
- `ProcessingJob` 数据类记录状态（pending → processing → completed/failed）
- 进度回调机制（`on_progress` 注册回调）
- 超时处理：30 分钟未完成的 processing 状态文档自动标记为 failed
- 前端通过 `GET /api/documents/{id}/status` 轮询进度

### 8.6 AI 知识提取（`knowledge_extraction_service.py`）

文档处理完成后异步执行两层提取：

1. **通用概念提取**（`extract_nodes_from_document`）
   - 分批处理 chunks（每批 3 个）
   - LLM 分析提取关键概念 → 生成 `KnowledgeNode` + `KnowledgeRelation`
   - 关联 `DocumentChunk.node_id` 建立 chunk → node 映射

2. **学习主线提取**（`extract_book_nodes`）
   - 提炼该书的 12 个专属学习主线节点
   - 供蜂巢技能树通关使用

### 8.7 知识库管理

- 用户可创建分类知识库（`KnowledgeBase`）
- 上传文档时关联到知识库
- 知识图谱支持按知识库 ID 过滤展示
- 知识提取生成的节点自动关联到对应知识库

## 11. 多 Agent 协同工作流

### 11.1 设计方案

基于 LangGraph `StateGraph` 编排三节点协同工作流：Orchestrator → RagBot → Reviewer，实现领域感知的智能教学。

### 11.2 工作流架构

```
用户消息
  ↓
Orchestrator (意图分析 + 领域分类 + 风格分类)
  → 单次 LLM 调用同时分类知识领域(domain)和教学风格(style)
  ↓
RagBot (混合检索)
  → BM25 关键词检索 + 向量语义检索 + RRF 融合
  ↓
Reviewer (最终回答生成)
  → 组合三段 prompt:
     1. 风格 prompt (HOW) — 从 Agent 表查询风格导师 system_prompt
     2. 领域 prompt (WHAT) — 从 DOMAINS 字典获取领域专属导师人设
     3. 认知状态 (针对谁) — 查询用户知识状态，注入薄弱点和已掌握情况
  → SSE 流式输出最终回答
```

### 11.3 AgentState 状态定义

```python
class AgentState(TypedDict, total=False):
    messages: list              # 对话历史
    user_id: str                # 用户ID
    sub_tasks: list             # Orchestrator 分解的子任务
    classified_domain: str      # 分类出的知识领域
    classified_style: str       # 分类出的教学风格
    agent_id: str               # 用户手动选择的导师 ID
    agent_results: dict         # 各 Agent 中间结果
    final_answer: str           # 最终聚合结果
    api_key: str                # LLM API Key
    model: str                  # 模型名
    base_url: Optional[str]     # 自定义 API 地址
    session: Any                # 数据库会话
    user_message: str           # 当前用户消息
```

`total=False` 允许字段渐进式填充，节点间通过读取和更新共享状态实现协同。

### 11.4 六大知识领域

| 领域 | 中文 | 导师人设 |
|---|---|---|
| programming | 编程开发基础 | Python 游戏与工具开发导师 |
| dsa | 数据结构与高级特性 | 益智游戏逻辑设计导师 |
| organization | 面向对象与系统架构 | OOP 构装设计导师 |
| os | 并发编程与操作系统 | 并发与系统编程导师 |
| network | 网络编程与联机服务 | 网络与 Web 编程导师 |
| database | 数据工程与持久化 | 数据持久化与工程实践导师 |

每个领域导师人设将 Python 概念与对应游戏场景结合（如贪吃蛇、2048、五子棋等）。

### 11.5 SSE 事件流

`POST /api/chat/graph` 返回 SSE 格式事件流：

## 13. 对话管理与上下文压缩

### 13.1 设计方案

对话服务实现无限轮对话，通过智能摘要和动态 Token 裁剪管理长对话上下文。

### 13.2 Token 管理

- **Token 计数**：`tiktoken` 精确计算，回退到 `len(text)//4` 估算
- **每条消息开销**：内容 Token + 4（消息边界开销）

### 13.3 智能摘要机制

触发条件：
- 消息数 >= 20 条
- 或累计 Token >= 100,000

摘要流程（`generate_summary`）：
1. 获取前 50% 的消息
2. 构建摘要 Prompt（每条消息截断 500 字符）
3. LLM 生成摘要
4. 更新 `Conversation.summary` + `summary_tokens`
5. 标记已摘要消息 `is_summarized=1`

### 13.4 优化上下文获取（`get_optimized_context`）

策略：
1. **摘要优先**：如有摘要，作为 system 消息添加
2. **最近消息**：倒序添加未摘要消息，直到达到 120,000 Token 预算
3. **跳过已摘要**：`is_summarized=1` 的消息不重复添加

### 13.5 后台任务

对话流式响应结束后触发两个后台任务：
1. **摘要生成**：`asyncio.create_task(conversation_service.generate_summary())`
2. **记忆提取**：`asyncio.create_task(_extract_memories_background())`（仅 `use_memory=True` 时）

## 16. 数据库设计与数据模型

### 16.1 数据库基础设施

- **引擎**: PostgreSQL 16 + pgvector 扩展
- **驱动**: asyncpg（URL 从 `postgresql://` 重写为 `postgresql+asyncpg://`）
- **ORM**: SQLAlchemy Async + `async_sessionmaker`
- **初始化**: `init_db()` 启用 pgvector 扩展 + 创建全部表 + ALTER TABLE 增量添加列

### 16.2 核心表结构

#### 用户与认证

| 表 | 模型 | 关键字段 |
|---|---|---|
| users | User | id(UUID), username(unique), hashed_password, role, nickname |
| agents | Agent | name, character_name, role_type, system_prompt, is_active |

#### 知识库与文档

| 表 | 模型 | 关键字段 |
|---|---|---|
| knowledge_bases | KnowledgeBase | name, description, owner_id |
| documents | Document | title, file_type, file_path, status, owner_id, visibility, is_active, knowledge_base_id |
| document_chunks | DocumentChunk | document_id, content, embedding(Vector), element_type, page_number, chunk_metadata(JSON), node_id |

#### 对话

| 表 | 模型 | 关键字段 |
|---|---|---|
| conversations | Conversation | user_id, agent_id, title, model, message_count, total_tokens, is_active(软删除), summary, summary_tokens |
| messages | Message | conversation_id, role, content, tokens, is_summarized |

#### 记忆

| 表 | 模型 | 关键字段 |
|---|---|---|
| memories | Memory | user_id, content, category, importance, source, embedding(Vector), access_count |
| memory_settings | MemorySetting | key(unique), value |

#### 知识图谱

| 表 | 模型 | 关键字段 |
|---|---|---|
| knowledge_nodes | KnowledgeNode | code, name, category, pagerank_weight, source(learning_path/extraction), document_id, knowledge_base_id |
| knowledge_relations | KnowledgeRelation | source_node_id, target_node_id, relation_type(requires/extends) |
| user_knowledge_states | UserKnowledgeState | user_id, node_id, proficiency, pagerank_score, is_lighted, study_duration |

#### 实验与练习

| 表 | 模型 | 关键字段 |
|---|---|---|
| labs | Lab | title, description, starter_code, test_cases(JSON), node_id, difficulty, lab_type, detailed_explanation |
| user_lab_submissions | UserLabSubmission | user_id, lab_id, submitted_code, status, evaluation_result(JSON), ai_feedback, score |
| user_collection_exercises | UserCollectionExercise | user_id, node_id, title, exercise_type, content(JSON), answer(JSON), explanation |

### 16.3 关系设计

- **级联删除**: User → Conversations/Memories/KnowledgeStates/Submissions（`cascade="all, delete-orphan"`）
- **软删除**: Conversation 通过 `is_active=0` 实现软删除
- **SET NULL**: Document.owner_id、KnowledgeNode.document_id 等使用 `ondelete="SET NULL"` 避免级联删除

### 16.4 向量列

两处使用 pgvector Vector 列：
- `document_chunks.embedding` — 文档分块向量
- `memories.embedding` — 记忆向量

通过 `cosine_distance` 函数进行相似度检索。

### 16.5 Schema 演进

项目使用 `ALTER TABLE ADD COLUMN IF NOT EXISTS` 实现增量 schema 演进（无 Alembic 迁移），在 `init_db()` 中执行 14 条 ALTER 语句添加新列。

### 18.2 环境变量

后端 `.env` 配置（`core/config.py` + Pydantic Settings）：
- `DATABASE_URL` — PostgreSQL 连接串
- `SECRET_KEY` — JWT 签名密钥
- `DEEPSEEK_API_KEY` / `DEEPSEEK_BASE_URL` / `DEEPSEEK_MODEL` — 系统级 LLM 配置

用户级 LLM 配置存储在前端 localStorage，请求时携带至后端。

### 18.3 CORS 配置

允许来源：`localhost:3000` / `localhost:3001` / `127.0.0.1:3000` / `127.0.0.1:3001`
- `allow_credentials=True`
- `expose_headers=["*"]`
- `max_age=86400`

### 18.4 本地嵌入模型部署

BGE-M3 通过 Ollama 本地部署：
1. 下载 BGE-M3 GGUF 模型文件
2. 创建 Ollama Modelfile
3. `ollama create bge-m3 -f Modelfile`
4. 验证：`curl http://localhost:11434/api/embeddings`

启用后在设置页开启本地嵌入开关，嵌入请求路由到 `http://localhost:11434`。

### 18.5 系统依赖

- **poppler**: PDF 渲染

Windows 环境使用 `python-magic-bin` 提供 DLL。

```
答题评测 → 通过?
  ├─ 是 → 点亮知识节点 + 提升熟练度 → PageRank 重算 → 推荐路径更新
```

```
上传文件 → unstructured 解析 → 语义分块 → 向量嵌入(API/本地BGE-M3) → pgvector 存储
                                                                        ↓
                                              AI 知识提取 → 注入知识图谱节点与关系
```

```
用户消息 → [use_rag?] 检索知识库向量 → [use_memory?] 检索长期记忆 → 构建 prompt

```
